# Contributing to WMS Platform

## Getting Started

```bash
git clone <repo-url>
cd wms-platform
./gradlew build        # Installs Git hooks automatically
```

The `build` task runs `installGitHooks`, which points Git to `.githooks/` and makes the scripts executable. Every developer who clones and builds gets hooks installed with zero manual steps.

---

## Branch Naming Convention

All branches must follow this format:

```
<type>/<service>/<author>/<description>
```

| Segment       | Allowed Values                                                                                                                                                                  |
|---------------|---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| **type**      | `feature`, `fix`, `hotfix`, `refactor`, `docs`, `test`, `chore`, `ci`, `perf`, `build`                                                                                         |
| **service**   | `identity-service`, `inventory-service`, `reservation-service`, `delivery-service`, `platform-service`, `api-gateway`, `config-server`, `eureka-server`, `wms-common`, `infra`, `all` |
| **author**    | lowercase alphanumeric + hyphens (`[a-z0-9-]+`)                                                                                                                                 |
| **description** | kebab-case (`[a-z0-9-]+`)                                                                                                                                                     |

**Examples:**

```
feature/identity-service/ali/add-oauth2-login
fix/reservation-service/jane/payment-race-condition
chore/infra/ali/upgrade-gradle-9
docs/all/ali/update-api-docs
```

**Exempt branches:** `main`, `master`, `develop`, `release/*`

## Commit Message Format

All commits must follow [Conventional Commits](https://www.conventionalcommits.org/):

```
<type>(<scope>): <description>
```

| Part          | Allowed Values                                                                              |
|---------------|---------------------------------------------------------------------------------------------|
| **type**      | `feat`, `fix`, `docs`, `refactor`, `test`, `chore`, `perf`, `ci`, `style`, `build`, `revert` |
| **scope**     | `identity`, `inventory`, `reservation`, `delivery`, `platform`, `gateway`, `common`, `infra`  |

**Rules:**
- Description starts with a **lowercase** letter
- Description does **not** end with a period
- First line is **72 characters or fewer**

**Examples:**

```
feat(identity): add OAuth2 login flow
fix(reservation): handle payment race condition
chore(infra): upgrade to Gradle 9.4
docs(common): update shared DTO javadoc
```

Merge commits and reverts generated by Git are allowed through automatically.

## Code Formatting

This project uses **Google Java Format** enforced by **Spotless**. Formatting is automatic at every stage — you should never need to think about it.

### How It Works

| Stage            | What Happens                                                        |
|------------------|---------------------------------------------------------------------|
| **On save (IDE)**| IntelliJ auto-formats the file using Google Java Format              |
| **On commit**    | `pre-commit` hook runs `spotlessApply` and re-stages formatted files |
| **On push**      | `pre-push` hook runs `spotlessCheck` — blocks push if not formatted  |

### IntelliJ Setup

The project ships with committed `.idea/` settings that enable:
- **Reformat code on save** — formats every file when you hit Ctrl+S / Cmd+S
- **Optimize imports on save** — removes unused imports automatically
- **Project code style** — configured to match Google Java Format

Spotless enforces exact Google Java Format on every commit and push, so any minor IDE drift gets caught automatically.

### Manual Commands

```bash
./gradlew spotlessApply    # Format all files
./gradlew spotlessCheck    # Check formatting (CI-friendly, no changes)
```

### What Gets Formatted

- **Java** (`*/src/**/*.java`) — Google Java Format (2-space indent, 100-char line)
- **Gradle Kotlin DSL** (`**/*.gradle.kts`) — ktlint

## Pull Request Workflow

1. Create a branch following the naming convention above
2. Make your changes and commit with proper messages
3. Push your branch and open a Pull Request
4. **Never push directly to `main`, `master`, or `develop`** — the `pre-push` hook blocks this
5. Exception: `hotfix/*` branches can push directly to protected branches for emergency fixes

---

## What the Git Hooks Enforce

| Hook          | What It Checks                                                                          |
|---------------|-----------------------------------------------------------------------------------------|
| `pre-commit`  | Branch name format, no secrets, debug statement warnings, no conflict markers, **auto-formats Java** |
| `commit-msg`  | Conventional Commits format, 72-char limit                                              |
| `pre-push`    | Blocks direct/force pushes to protected branches, **spotlessCheck**, `compileJava`      |

Hooks are installed automatically via `./gradlew build`. To reinstall manually:

```bash
./gradlew installGitHooks
```
